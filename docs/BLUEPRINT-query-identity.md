# API Almanac Reference and Identity Scheme

This document describes the proposed identity, naming, and reference scheme for API Almanac request files.

The goal is to preserve the strengths of an Obsidian-like, local-first project structure while also making references robust under common operations such as renaming, moving, importing, merging, and LLM-assisted editing.

API Almanac projects should remain pleasant to browse as ordinary folders and text files. At the same time, the application needs durable internal identities so that one request can safely refer to values produced by another request.

The proposed solution is a hybrid scheme:

```text
{{class:uid:semantic-id:value-path}}
```

For request-to-request references:

```text
{{req:U4ABS6QK:authentication.plain_login:reply.token}}
```

This gives us four pieces of information:

```text
req                 reference class / namespace
U4ABS6QK            stable compact UID
authentication.plain_login
                    current semantic slug-like ID
reply.token         selected value path
```

The UID is authoritative. The semantic ID exists for readability, stale-reference detection, and repair. The value path selects the specific output, field, or named value being referenced.

---

## Design Goals

The scheme should work well for four audiences:

1. **The GUI** — must resolve references reliably, offer friendly pickers, survive renames and moves, and detect stale labels.
2. **Humans reading text files** — should be able to understand the project without opening the GUI.
3. **LLMs reading or generating files** — should be able to infer API workflows, authentication dependencies, and request relationships from raw files.
4. **Programmatic tools** — should be able to parse, validate, refactor, grep, index, and rewrite references reliably.

The design deliberately avoids using opaque UUIDs as the only visible identity. It also avoids making file paths the sole source of truth, because file paths change too easily.

---

## Core Concepts

Each request has three separate names or identities:

```yaml
uid: U4ABS6QK
id: authentication.plain_login
name: Plain Login
```

These serve different roles.

### `uid`

The `uid` is the permanent identity of the request.

Example:

```yaml
uid: U4ABS6QK
```

Properties:

* Generated by the application.
* Stable across renames and moves.
* Short enough to be readable in text files.
* Used as the authoritative reference target.
* Should not normally be edited by users.
* May be regenerated during imports if a collision is detected.

This is conceptually similar to a UUID, but much shorter and more pleasant in raw text.

Recommended format:

```text
[A-Z0-9]{8}
```

Example:

```text
U4ABS6QK
Q9PZ11AH
B72KMA90
```

Eight base-36 characters provide a very large space while remaining compact.

### `id`

The `id` is the semantic, slug-like identity.

Example:

```yaml
id: authentication.plain_login
```

Properties:

* Human-readable.
* LLM-readable.
* Derived from the request's semantic location in the project.
* Used inside references for legibility.
* Can be automatically updated when files are renamed or moved.
* Not authoritative for reference resolution when the UID is present.

The semantic ID should normally mirror the group/path structure:

```text
authentication/plain-login.request.md
```

becomes:

```yaml
id: authentication.plain_login
```

A deeper path such as:

```text
admin-api/authentication/plain-login.request.md
```

becomes:

```yaml
id: admin_api.authentication.plain_login
```

### `name`

The `name` is the GUI/display name.

Example:

```yaml
name: Plain Login
```

Properties:

* User-facing.
* May contain spaces, punctuation, uppercase letters, and friendly wording.
* May duplicate names in other groups.
* Is not used as a durable reference identity.

This should be allowed:

```text
Public API / Authentication
Admin API / Authentication
Mobile API / Authentication
```

All three may have:

```yaml
name: Authentication
```

but they should have different `uid` and `id` values.

## Reference Format

The canonical reference payload is:

```text
class:uid:semantic-id:value-path
```

Usually wrapped in interpolation braces:

```text
{{class:uid:semantic-id:value-path}}
```

For request references:

```text
{{req:U4ABS6QK:authentication.plain_login:reply.token}}
```

### Field 1: `class`

The class tells the resolver what kind of object is being referenced.

Examples:

```text
req
env
secret
var
```

For request-to-request references, the class is:

```text
req
```

### Field 2: `uid`

The UID is the stable target identity.

Example:

```text
U4ABS6QK
```

For request references, this should match the request file's `uid` field.

### Field 3: `semantic-id`

The semantic ID is the current readable identifier.

Example:

```text
authentication.plain_login
```

The resolver should not depend on this when the UID is valid, but it should use it to detect stale references.

For example, if a reference says:

```text
{{req:U4ABS6QK:authentication.plain_login:reply.token}}
```

but the target request now says:

```yaml
uid: U4ABS6QK
id: authentication.login
```

then the reference still resolves, but the semantic ID is stale. The application can offer to rewrite it as:

```text
{{req:U4ABS6QK:authentication.login:reply.token}}
```

### Field 4: `value-path`

The value path selects what value is being used from the target object.

Examples:

```text
reply.token
reply.user_id
reply.order.id
body.customer.email
```

For requests, this should normally point to a named extracted value from the response, not necessarily directly to arbitrary raw JSON.

For example:

```yaml
reply:
  token:
    from: response.body.authorization.token
    description: Bearer token returned by login.
```

Then another request can use:

```text
{{req:U4ABS6QK:authentication.plain_login:reply.token}}
```

This is preferable to forcing all consumers to know the exact response JSON path.

---

## Separator Rules

The colon `:` is reserved as the top-level separator.

References are exactly four fields:

```text
class:uid:semantic-id:value-path
```

A parser can split on `:` and expect exactly four parts.

To preserve this simplicity:

* `:` must not appear inside any field.
* Escaping should not be supported initially.
* Display names may contain colons, but semantic IDs may not.
* The semantic ID and value path use `.` for hierarchy.

Recommended grammar:

```text
class       = [a-z][a-z0-9_-]*
uid         = [A-Z0-9]{8}
semantic-id = [a-z0-9_]+(\.[a-z0-9_]+)*
value-path  = [a-z0-9_]+(\.[a-z0-9_]+)*
```

Examples:

```text
{{req:U4ABS6QK:authentication.plain_login:reply.token}}
{{req:Q9PZ11AH:orders.create_order:reply.order_id}}
{{req:B72KMA90:users.lookup_by_email:reply.user.id}}
```

## Example Request File

```markdown
---
type: request
uid: U4ABS6QK
id: authentication.plain_login
name: Plain Login
description: Logs in with username and password and returns a bearer token.
method: POST
url: "{{env:LOCAL001:local:base_url}}/auth/login"

headers:
  Content-Type: application/json

body:
  type: json
  value:
    username: "{{secret:ADMIN001:admin:username}}"
    password: "{{secret:ADMIN001:admin:password}}"

reply:
  token:
    from: response.body.authorization.token
    description: Bearer token returned by the plain login endpoint.

  user_id:
    from: response.body.user.id
    description: ID of the authenticated user.
---

# Plain Login

This request obtains an authorization token using a username and password.

The most commonly reused value is `reply.token`.
```

Another request might use it as:

```yaml
headers:
  Authorization: "Bearer {{req:U4ABS6QK:authentication.plain_login:reply.token}}"
```

The GUI can render this as a friendly chip:

```text
Bearer [Authentication / Plain Login → token]
```

while the raw file remains meaningful:

```text
Bearer {{req:U4ABS6QK:authentication.plain_login:reply.token}}
```

---

## Why This Is Good for the GUI

The GUI gets a reliable, durable reference mechanism.

When rendering a reference, the GUI can:

1. Parse the four fields.
2. Use the class to select the resolver.
3. Use the UID to find the target request.
4. Check whether the semantic ID is current.
5. Resolve the value path.
6. Render a friendly display label.

The GUI does not have to break references when users rename or move files.

For example:

```text
{{req:U4ABS6QK:authentication.plain_login:reply.token}}
```

can be displayed as:

```text
Plain Login → token
```

or, with group context:

```text
Authentication / Plain Login → token
```

or as a chip:

```text
[Authentication / Plain Login → token]
```

The GUI can also detect stale semantic labels:

```text
Reference still points to UID U4ABS6QK, but its semantic ID changed from authentication.plain_login to authentication.login.
```

Then it can offer:

```text
Update references
Ignore
Show target request
```

This makes references both robust and explainable.

---

## Why This Is Good for Humans Reading Text Files

A human opening the project in a text editor sees meaningful information:

```text
{{req:U4ABS6QK:authentication.plain_login:reply.token}}
```

Even without knowing what `U4ABS6QK` means, the human can infer:

```text
This uses the token from the plain login request in the authentication area.
```

The text file is not polluted with long UUIDs as the only readable identity.

Compare:

```text
{{req:7f3a4e21-8a9b-4c13-9a2d-9918f46f61d3:reply.token}}
```

versus:

```text
{{req:U4ABS6QK:authentication.plain_login:reply.token}}
```

The second is much better for local-first, Git-friendly, Obsidian-like usage.

---

## Why This Is Good for LLMs Reading the Files

LLMs benefit from semantic redundancy.

A reference like this:

```text
{{req:U4ABS6QK:authentication.plain_login:reply.token}}
```

contains enough meaning for an LLM to infer the workflow:

* There is an authentication-related request.
* It performs a plain login.
* It produces a token.
* Another request depends on that token.

The LLM does not need access to the GUI or an internal database to understand the project.

This matters because API Almanac is intended to support workflows such as:

* An LLM reads API documentation and generates a starter project.
* An LLM reads an existing API Almanac project and learns how to call the API.
* An LLM modifies request files, creates new examples, or adds missing tests.
* An LLM repairs broken project structure after a refactor.

The UID protects against accidental breakage. The semantic ID preserves legibility.

This is a deliberate form of redundancy. It is not wasteful; it is useful.

---

## Why This Is Good for Programmatic Tools

Programmatic tools get a simple, regular format.

A tool can scan files for:

```text
{{...}}
```

Then parse reference payloads by splitting on `:`.

Because each reference has exactly four fields, tools can easily validate them:

```text
class:uid:semantic-id:value-path
```

A tool can build indexes:

```text
uid → file path
semantic-id → uid
old alias → uid
file path → uid
```

A tool can perform safe refactors:

* Rename semantic IDs.
* Rewrite references.
* Detect stale references.
* Find unresolved UIDs.
* Detect duplicate UIDs.
* Detect duplicate semantic IDs.
* Generate dependency graphs.
* Extract authentication flows.
* Validate that referenced `reply` fields exist.

The format is grep-friendly:

```bash
grep -R "req:U4ABS6QK" .
grep -R "authentication.plain_login" .
grep -R "reply.token" .
```

It is also friendly to simple scripts in Python, Rust, JavaScript, shell, or other tools.

## Creating a Request

When a new request is created, the application should generate:

1. A stable UID.
2. A semantic ID derived from the target folder and request title.
3. A display name.
4. A filename derived from the display name or final semantic segment.

Example user action:

```text
Create request "Plain Login" inside group "Authentication"
```

Project path:

```text
authentication/plain-login.request.md
```

Generated metadata:

```yaml
uid: U4ABS6QK
id: authentication.plain_login
name: Plain Login
```

If the semantic ID already exists, append a numeric suffix:

```yaml
id: authentication.plain_login_2
```

The filename may similarly become:

```text
plain-login-2.request.md
```

or:

```text
plain-login_2.request.md
```

The exact filename convention is less important than consistency.

Recommended rule:

* Use hyphens in filenames.
* Use underscores in semantic IDs.
* Use dots between semantic hierarchy segments.

Example:

```text
File: authentication/plain-login.request.md
ID:   authentication.plain_login
```

---

## Deleting a Request

Deleting a request should be treated carefully because other requests may depend on it.

Before deletion, the app should search all project files for references to the request UID.

Example target:

```yaml
uid: U4ABS6QK
id: authentication.plain_login
```

References to find:

```text
{{req:U4ABS6QK:authentication.plain_login:reply.token}}
```

If references exist, the GUI should warn:

```text
This request is referenced by 3 other requests.
Deleting it will break those references.
```

Possible actions:

1. Cancel deletion.
2. Delete anyway and mark references as unresolved.
3. Replace references with another request's value.
4. Comment out or preserve broken references for manual repair.

The raw reference remains meaningful even after deletion. A human or LLM can see what used to be referenced:

```text
{{req:U4ABS6QK:authentication.plain_login:reply.token}}
```

The resolver can report:

```text
Unresolved request UID U4ABS6QK, formerly labeled authentication.plain_login.
```

This is much better than losing all context.

## Renaming a Request

There are two kinds of rename.

### Display Rename

A display rename changes only the user-facing name.

Before:

```yaml
uid: U4ABS6QK
id: authentication.plain_login
name: Plain Login
```

After:

```yaml
uid: U4ABS6QK
id: authentication.plain_login
name: Login
```

References do not need to change.

This is the safest operation and should be the default for simple title edits.

### Semantic Rename

A semantic rename changes the request's slug-like ID and optionally the filename.

Before:

```yaml
uid: U4ABS6QK
id: authentication.plain_login
name: Plain Login
```

After:

```yaml
uid: U4ABS6QK
id: authentication.login
name: Login
aliases:
  - authentication.plain_login
```

References should be automatically rewritten:

Before:

```text
{{req:U4ABS6QK:authentication.plain_login:reply.token}}
```

After:

```text
{{req:U4ABS6QK:authentication.login:reply.token}}
```

If some references are missed, they still resolve through the UID. The semantic ID is stale but repairable.

The GUI can offer an operation such as:

```text
Rename semantically and update references
```

This operation may update:

* `name`
* `id`
* filename
* references in other files
* aliases

## Moving a Request

Moving a request changes its location in the directory tree.

Before:

```text
authentication/plain-login.request.md
```

```yaml
uid: U4ABS6QK
id: authentication.plain_login
```

After moving to:

```text
admin-api/authentication/plain-login.request.md
```

there are two possible policies.

### Physical Move Only

The file moves, but the semantic ID remains unchanged:

```yaml
uid: U4ABS6QK
id: authentication.plain_login
```

References do not need to change.

This is safe and conservative.

### Semantic Move

The file moves and the semantic ID is updated to reflect the new location:

```yaml
uid: U4ABS6QK
id: admin_api.authentication.plain_login
aliases:
  - authentication.plain_login
```

References should be automatically rewritten:

Before:

```text
{{req:U4ABS6QK:authentication.plain_login:reply.token}}
```

After:

```text
{{req:U4ABS6QK:admin_api.authentication.plain_login:reply.token}}
```

Again, stale references remain resolvable because the UID is stable.

Recommended UX:

```text
Move request
[ ] Also update semantic ID and references
```

or two separate commands:

```text
Move file
Move and refactor identity
```

The second option is better when the move represents conceptual reorganization, not merely file housekeeping.

## Reconstructing Semantic IDs from the Directory Tree

The application should be able to reconstruct suggested semantic IDs from the directory structure.

Example path:

```text
groups/admin-api/authentication/plain-login.request.md
```

Remove project-specific container directories such as:

```text
groups/
```

Remove the request extension:

```text
.request.md
```

Convert path segments into slug segments:

```text
admin-api
authentication
plain-login
```

Normalize each segment:

```text
admin_api
authentication
plain_login
```

Join with dots:

```text
admin_api.authentication.plain_login
```

The app can compare this reconstructed ID to the file's current ID:

```yaml
id: authentication.plain_login
```

If they differ, the app can report:

```text
Current ID:   authentication.plain_login
Suggested ID: admin_api.authentication.plain_login
```

Possible actions:

1. Keep current ID.
2. Update semantic ID and rewrite references.
3. Add suggested ID as an alias.
4. Rename/move file to match current ID.

This helps keep the project clean without making paths dangerously authoritative.

## Naming Conflicts

There are three different kinds of conflicts.

### Display Name Conflicts

Display name conflicts are allowed.

This is fine:

```text
Admin API / Authentication
Public API / Authentication
```

Both may have:

```yaml
name: Authentication
```

The GUI should disambiguate them using group context.

### Semantic ID Conflicts

Semantic ID conflicts are not ideal and should be resolved.

Example conflict:

```yaml
id: authentication.plain_login
```

already exists.

A newly created or imported request also wants:

```yaml
id: authentication.plain_login
```

Resolve by appending a numeric suffix:

```yaml
id: authentication.plain_login_2
```

If that exists:

```yaml
id: authentication.plain_login_3
```

The suffix should be appended to the last segment, not the whole dotted path.

Good:

```text
authentication.plain_login_2
```

Less good:

```text
authentication.plain_login._2
```

Also acceptable when context exists:

```text
admin_api.authentication.plain_login
public_api.authentication.plain_login
```

The resolver should use UID first, so semantic ID collisions are repairable, but the project should still try to keep semantic IDs unique.

### UID Conflicts

UID conflicts are serious and should be rare.

If two request files contain:

```yaml
uid: U4ABS6QK
```

then the app must resolve the collision.

Possible causes:

* A request was duplicated manually.
* A project was copied and merged.
* An LLM generated files using an existing UID.
* A rare random collision occurred.

Recommended behavior:

1. Detect duplicate UIDs during project indexing.
2. Determine whether the files are exact duplicates or distinct requests.
3. If distinct, assign a new UID to one of them.
4. Rewrite references inside the affected imported/copied subtree when possible.
5. Preserve old UID as an alias only if useful and safe.

Example:

Before:

```yaml
uid: U4ABS6QK
id: authentication.plain_login
```

Conflicting imported file becomes:

```yaml
uid: V9KD2PZA
id: authentication.plain_login_2
aliases:
  - authentication.plain_login
```

References inside the imported subtree should be updated from:

```text
{{req:U4ABS6QK:authentication.plain_login:reply.token}}
```

to:

```text
{{req:V9KD2PZA:authentication.plain_login_2:reply.token}}
```

But references outside the imported subtree should continue pointing to the original request unless explicitly changed.

## Aliases

Aliases preserve knowledge of previous semantic IDs.

Example:

```yaml
uid: U4ABS6QK
id: authentication.login
aliases:
  - authentication.plain_login
  - auth.old_login
```

Aliases are useful for:

* Detecting stale references.
* Repairing references after manual edits.
* Helping LLMs understand historical names.
* Supporting Git merges.
* Making refactors less brittle.

Aliases should not become the primary resolution mechanism when UIDs are present. They are mainly for diagnostics and repair.

If a reference has a valid UID but stale semantic ID, the app should resolve by UID and optionally update the reference.

If a reference is missing or has an invalid UID but its semantic ID matches an alias, the app may suggest a repair:

```text
Reference could not be resolved by UID, but semantic ID authentication.plain_login is an alias of authentication.login.
```

Then the app can offer to rewrite the reference with the current UID and semantic ID.

## Reference Resolution Algorithm

For a request reference:

```text
{{req:U4ABS6QK:authentication.plain_login:reply.token}}
```

Recommended resolution algorithm:

1. Parse the reference into exactly four fields.
2. Confirm `class == req`.
3. Look up request by UID `U4ABS6QK`.
4. If found:

   * Check whether the request's current `id` equals `authentication.plain_login`.
   * If not, mark semantic ID as stale.
   * Resolve `reply.token` against the target request.
5. If UID is not found:

   * Try exact semantic ID match.
   * Try alias match.
   * Try fuzzy/path-based suggestions.
   * Mark unresolved if no candidate exists.
6. If value path does not exist:

   * Show target request but mark selected value as missing.
   * Suggest nearby value paths if possible.

This gives graceful degradation.

A broken reference still carries useful information:

```text
class: req
old UID: U4ABS6QK
old semantic ID: authentication.plain_login
wanted value: reply.token
```

That is enough for humans, tools, or LLMs to attempt repair.

---

## Automatic Reference Updates

Whenever the application performs a semantic rename or semantic move, it should update references across the project.

Example old reference:

```text
{{req:U4ABS6QK:authentication.plain_login:reply.token}}
```

New request metadata:

```yaml
uid: U4ABS6QK
id: authentication.login
```

Updated reference:

```text
{{req:U4ABS6QK:authentication.login:reply.token}}
```

The UID stays the same.

If the update misses a reference, it is not catastrophic because the UID still resolves.

The application can periodically run a repair pass:

```text
Find stale reference labels
Update stale reference labels
Validate all references
```

This is especially important after:

* Git merges.
* LLM-generated edits.
* Manual text editing.
* Bulk file moves.
* Imports from another project.

---

## Importing and Merging Projects

When importing another API Almanac project, the app should index both projects and detect:

* Duplicate UIDs.
* Duplicate semantic IDs.
* File path conflicts.
* Broken references.

Recommended import behavior:

1. Preserve UIDs when there is no conflict.
2. Preserve semantic IDs when there is no conflict.
3. If a semantic ID conflicts, append `_{n}` to the imported request's final segment.
4. If a UID conflicts and the files are not the same logical request, regenerate the imported UID.
5. Rewrite references inside the imported subtree to match regenerated UIDs and semantic IDs.
6. Leave references to external project objects unresolved or explicitly mark them as external.

Example semantic conflict:

```yaml
id: authentication.plain_login
```

becomes:

```yaml
id: authentication.plain_login_2
```

Example UID conflict:

```yaml
uid: U4ABS6QK
```

becomes:

```yaml
uid: V9KD2PZA
```

Then internal imported references are rewritten accordingly.

---

## LLM-Generated Projects

LLMs should be encouraged to generate both semantic IDs and placeholder UIDs.

However, because LLMs may accidentally duplicate UIDs or invent invalid ones, the app should validate and repair generated projects on first load.

Recommended instruction to LLMs:

```text
Generate readable semantic IDs from the directory structure.
Generate unique 8-character uppercase alphanumeric UIDs for each request.
Use the reference format {{req:UID:semantic.id:reply.field}}.
```

On load, the app validates:

* UID format.
* UID uniqueness.
* semantic ID uniqueness.
* reference parseability.
* reference target existence.
* value path existence.

If the LLM generated weak or duplicate UIDs, the app can repair them while preserving semantic structure.

## Recommended Project Hygiene Commands

API Almanac should eventually expose maintenance commands such as:

```text
Validate Project
Repair Stale References
Align IDs With File Paths
Align File Paths With IDs
Regenerate Duplicate UIDs
Find Broken References
Show Dependency Graph
```

These commands are especially valuable because the project is intentionally editable outside the GUI.

### Validate Project

Checks that all request files have valid metadata and that all references resolve.

### Repair Stale References

Rewrites semantic IDs inside references when the UID still points to a valid request.

### Align IDs With File Paths

Computes suggested semantic IDs from the directory structure and updates request IDs, aliases, and references.

### Align File Paths With IDs

Moves or renames files to match current semantic IDs.

### Regenerate Duplicate UIDs

Fixes UID collisions and updates affected references.

### Find Broken References

Reports unresolved UIDs, missing value paths, and malformed references.

### Show Dependency Graph

Displays which requests depend on values produced by other requests.

## Recommended Rules Summary

1. Every request has a stable `uid`.
2. Every request has a semantic `id`.
3. Every request has a user-facing `name`.
4. The `uid` is authoritative for references.
5. The semantic `id` is included in references for readability and repair.
6. The display `name` may duplicate names elsewhere.
7. References use exactly four colon-separated fields.
8. `:` is reserved and cannot appear inside reference fields.
9. `.` separates semantic path segments and value path segments.
10. Renaming or moving does not change the UID.
11. Semantic renames and moves should rewrite references.
12. Stale references should still resolve if the UID is valid.
13. Semantic ID conflicts are resolved by appending `_{n}`.
14. UID conflicts are resolved by regenerating one UID and rewriting affected references.
15. The directory tree remains semantically meaningful, but not dangerously authoritative.

---

## Final Conceptual Model

The directory tree is the semantic map.

The YAML frontmatter is the semantic contract.

The UID is the durable anchor.

The semantic ID is the readable label.

The display name is the GUI title.

The reference format carries both machine stability and human meaning:

```text
{{req:U4ABS6QK:authentication.plain_login:reply.token}}
```

This is robust enough for the GUI, readable enough for humans, meaningful enough for LLMs, and regular enough for programmatic tools.
